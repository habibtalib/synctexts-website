## Summary

Phase 5 extends the existing Drizzle/better-sqlite3/SQLite stack with 8 nullable columns on the `submissions` table, establishes the generate+migrate production workflow (replacing any push usage), and hardens the DB connection with WAL mode and busy_timeout.

The existing codebase already has drizzle-orm 0.40.1, drizzle-kit 0.31.9, and better-sqlite3 11.10.0 installed. No new packages are needed. The `drizzle/` migrations directory does not yet exist — `db:generate` will create it on first run.

The WAL PRAGMA must be set on the raw `better-sqlite3` connection object before calling `drizzle()`, not through migration files. The `db:migrate` script must shell-copy the `.db` file before calling `drizzle-kit migrate` — drizzle-kit has no built-in backup feature.

**Primary recommendation:** Set PRAGMAs on the raw `sqlite` instance before `drizzle()` wraps it; use a `predb:migrate` npm script hook or inline shell command for backup; generate migrations with drizzle-kit and apply with drizzle-kit.

### Pattern 1: Schema Extension with Nullable Columns

**What:** Add new columns to existing table as nullable (or with .default() for lead_status). Drizzle-kit generate detects the diff against the last snapshot and emits `ALTER TABLE ... ADD COLUMN` SQL.

**When to use:** Any time columns are added to an existing production table that has rows.

**Key rule:** `ALTER TABLE ... ADD COLUMN` in SQLite can add nullable columns or columns with constant defaults without rewriting the table. This is safe on production data.

```typescript
// src/db/schema.ts — Source: drizzle-orm SQLite column types docs
import { sqliteTable, text, integer, real } from 'drizzle-orm/sqlite-core';

export const submissions = sqliteTable('submissions', {
  // ... existing 8 columns unchanged ...

  // New columns — Phase 5
  serviceType: text('service_type'),                         // nullable
  budget: text('budget'),                                    // nullable
  timeline: text('timeline'),                                // nullable
  leadScore: integer('lead_score'),                          // nullable (integer)
  leadStatus: text('lead_status').notNull().default('new'),  // NOT NULL with DB default
  notes: text('notes'),                                      // nullable
  hubspotId: text('hubspot_id'),                             // nullable
  hubspotSyncedAt: text('hubspot_synced_at'),                // nullable (ISO string)
});
```

**Important:** `leadStatus` uses `.default('new')` (not `.$defaultFn()`). This emits a SQL-level `DEFAULT 'new'`, which means SQLite applies it when existing rows are encountered via `ALTER TABLE ADD COLUMN` — all existing rows immediately get `lead_status = 'new'` without an UPDATE statement.

### Pattern 2: WAL Mode + PRAGMA in db/index.ts

**What:** Execute PRAGMAs on the raw `better-sqlite3` connection before the `drizzle()` wrapper is created.

**When to use:** Always — WAL mode is per-connection and must be set at startup.

```typescript
// src/db/index.ts — Source: drizzle-orm/better-sqlite3 GitHub issue #4968
import Database from 'better-sqlite3';
import { drizzle } from 'drizzle-orm/better-sqlite3';
import * as schema from './schema';
import path from 'node:path';
import { mkdirSync } from 'node:fs';

const dbDir = path.join(process.cwd(), 'data');
mkdirSync(dbDir, { recursive: true });

const dbPath = path.join(dbDir, 'submissions.db');
const sqlite = new Database(dbPath);

// Set WAL mode and concurrency settings BEFORE drizzle() wraps the connection
sqlite.pragma('journal_mode = WAL');
sqlite.pragma('busy_timeout = 5000');
sqlite.pragma('synchronous = NORMAL');   // Safe with WAL; improves write throughput

export const db = drizzle({ client: sqlite, schema });
// Note: sqlite is NOT exported — only db
```

**Critical:** `journal_mode = WAL` is persistent (stored in the database file) once set, but `busy_timeout` is per-connection and must be set every time a connection is opened. Both must appear before `drizzle()` init.

### Pattern 3: Migration Workflow npm Scripts

**What:** A three-script setup where `db:migrate` backs up the database before calling drizzle-kit.

```json
{
  "scripts": {
    "db:generate": "drizzle-kit generate",
    "db:migrate": "cp data/submissions.db data/submissions.db.bak && drizzle-kit migrate",
    "db:studio": "drizzle-kit studio"
  }
}
```

**Notes:**
- `cp` works on macOS (production target). For cross-platform safety, could use `node -e "require('fs').cpSync(...)"` but `cp` is fine for this project.
- The `&&` operator ensures migration only runs if backup succeeds.
- `drizzle-kit push` must be completely removed from scripts. It does not appear in the current `package.json` — confirm it is absent before shipping.

### Pattern 4: First-Time Migration (bootstrap)

Because the `drizzle/` folder does not yet exist, the first `db:generate` run will:
1. Read `src/db/schema.ts`
2. Create `drizzle/meta/_journal.json` and a snapshot
3. Emit `drizzle/0000_<name>.sql` with the full current schema (all 16 columns)

Running `db:migrate` after will:
1. Copy `data/submissions.db` to `data/submissions.db.bak`
2. Create `__drizzle_migrations` tracking table
3. Apply the SQL — since the `submissions` table already exists with 8 columns, the generated SQL must use `ALTER TABLE ADD COLUMN` for the 8 new columns, not `CREATE TABLE`

**Risk:** If drizzle-kit generates a `CREATE TABLE` for the initial snapshot, it will fail on the existing table. Verify the generated SQL contains `ALTER TABLE submissions ADD COLUMN` for each new column, not a `CREATE TABLE submissions`. This is the expected behavior when `db:generate` runs against a database that already has the table, but confirm by inspecting `0000_*.sql` before running `db:migrate`.

### Anti-Patterns to Avoid

- **`drizzle-kit push` on production:** Bypasses migration tracking, can silently drop columns. Must not appear in any npm script.
- **PRAGMA in migration SQL files:** WAL mode set via a `.sql` migration file does not reliably activate — must be on the raw connection at startup.
- **Exporting the raw `sqlite` instance:** Only `db` should be exported. Raw instance leaks implementation detail and enables bypassing Drizzle type safety.
- **Auto-running migrations on app startup:** `migrate()` called at Astro startup creates race conditions in multi-instance scenarios and blocks startup time.
- **`.$defaultFn()` for lead_status:** Uses application-level default only, does not emit `DEFAULT 'new'` in SQL. Existing rows would get NULL. Must use `.default('new')` for the SQL-level default to backfill existing rows.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Schema diff + SQL generation | Custom diff script | `drizzle-kit generate` | Handles renames, constraint changes, type promotions, snapshot tracking |
| Migration tracking (which ran) | Custom `__migrations` table | `drizzle-kit migrate` | Creates and manages `__drizzle_migrations` internally |
| SQL backup before migration | Nothing extra needed | `cp file file.bak && drizzle-kit migrate` | One-liner; SQLite is a single file |

**Key insight:** SQLite's single-file nature makes backup trivial — `cp` before migrate is the entire backup strategy. No streaming backup, no pg_dump equivalent needed.

## Common Pitfalls

### Pitfall 1: CREATE TABLE vs ALTER TABLE in First Migration

**What goes wrong:** When `drizzle/` is created for the first time against an existing database, drizzle-kit may generate a `CREATE TABLE submissions` statement (full schema), which will error because the table exists.

**Why it happens:** drizzle-kit has no connection to the live DB during `generate` — it compares snapshots. First run has no previous snapshot to compare against, so it may emit full CREATE TABLE.

**How to avoid:** After running `db:generate`, open `drizzle/0000_*.sql` and verify. If it contains `CREATE TABLE submissions`, you have two options:
1. Manually edit the SQL to replace with 8 `ALTER TABLE submissions ADD COLUMN` statements.
2. Run `drizzle-kit push` once to sync state (on a dev copy only), then `generate` against the synced DB to get a clean migration with only the diff.

**Warning signs:** The generated SQL file contains `CREATE TABLE` with all 16 columns instead of 8 `ADD COLUMN` statements.

**Resolution path:** The safest approach is to test `db:migrate` against a copy of the database first, then run on the real file once the SQL is confirmed correct.

### Pitfall 2: busy_timeout vs busy_handler

**What goes wrong:** Using `sqlite.pragma('busy_timeout = 5000')` on better-sqlite3 is correct, but some older docs show `Database({ timeout: 5000 })` as constructor option.

**Why it happens:** The constructor `timeout` option is the busy timeout, but it is better-sqlite3's abstraction. The PRAGMA `busy_timeout` is the SQLite-native way and takes milliseconds.

**How to avoid:** Use `sqlite.pragma('busy_timeout = 5000')` — this is equivalent to SQLite's `sqlite3_busy_timeout(db, 5000)`. Both set a 5-second wait before returning SQLITE_BUSY. The PRAGMA approach is more explicit and matches the project decisions verbatim.

### Pitfall 3: WAL Mode on First Open

**What goes wrong:** `journal_mode = WAL` is a persistent setting once set, but if the `.db` file was created in the default DELETE journal mode and you add the PRAGMA later, the first connection after the change writes the WAL header. Concurrent connections that opened before the PRAGMA ran won't see WAL mode.

**Why it happens:** WAL mode change takes effect immediately for the connection that issues it, and persists in the DB file for future connections.

**How to avoid:** In this project there is only one Node.js process (Astro SSR), opened fresh on each deploy. The PRAGMA in `db/index.ts` runs before any queries. Not a real concern here, but worth knowing.

### Pitfall 4: .db.bak Committed to Git

**What goes wrong:** `data/submissions.db.bak` gets committed accidentally.

**Why it happens:** `data/` may not be in `.gitignore`.

**How to avoid:** Verify `.gitignore` includes `data/` or at minimum `*.db` and `*.db.bak` before running `db:migrate`.

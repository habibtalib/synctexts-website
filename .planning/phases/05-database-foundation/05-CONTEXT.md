<decisions>
## Implementation Decisions

### Budget & Timeline Values
- Budget stored as TEXT with 4 predefined values: `under_5k`, `5k_15k`, `15k_50k`, `50k_plus`
- Timeline stored as TEXT with 4 predefined values: `asap`, `1_3_months`, `3_6_months`, `exploring`
- Service type stored as TEXT with 3 values: `web_dev`, `devops`, `analytics` (no "other" option)
- Lead status defaults to `'new'` for all submissions (not NULL) ‚Äî values: `new`, `contacted`, `qualified`, `proposal_sent`, `won`, `lost`

### Migration Safety
- npm scripts for migration workflow: `db:generate`, `db:migrate`, `db:studio`
- `db:migrate` automatically backs up `data/submissions.db` to `.bak` before running `drizzle-kit migrate`
- Migrations run manually only ‚Äî never auto-run on app startup
- Existing v1.0 submissions get `lead_status = 'new'` via column default during migration
- Existing v1.0 submissions keep `lead_score = NULL` (no data to score ‚Äî dashboard shows 'N/A')

### DB Initialization
- WAL mode and `busy_timeout = 5000` set in `src/db/index.ts` at app startup (PRAGMA statements after Database creation)
- Only export Drizzle wrapper (`db`) ‚Äî no raw sqlite instance export

### Claude's Discretion
- Exact migration SQL generated by drizzle-kit
- Column ordering in schema definition
- Any additional PRAGMAs for performance (e.g., `synchronous = NORMAL`)

</decisions>
